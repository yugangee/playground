## 1. 개요

이 문서는 PLAYGROUND 플랫폼의 대회 관리 기능을 상세히 정의한다. 기존 `league` Lambda가 제공하는 리그/토너먼트 기본 기능 위에, 프로 수준의 대진표 시각화, 쿼터별 라인업 전술판, 실시간 경기 기록, 누적 통계 대시보드, 카카오톡 연동 공지를 추가한다.

**핵심 사용자 시나리오:**

- 총무(운영자): 대회 생성 → 팀 등록 → 대진표 자동 생성 → 라인업 편집 → 경기 실시간 기록 → 결과 확정 → 통계 공유
- 선수(팀원): 카카오톡 카드로 소집 확인 → 투표 → 라인업 확인 → 경기 후 스탯 조회 → MOM 투표

```

    maxGuestsPerMatch: number         // 경기당 최대 용병 수
  }
  registration: {
    visibility: "public" | "private"
    entryFee: number | null
    minTeams: number
    maxTeams: number
    registrationDeadline: ISO8601
  }
  venue: {
    name: string
    address: string
    parkingInfo: string | null
    mapUrl: string | null             // 카카오맵 링크
  }
  awards: string[]                    // ["우승", "준우승", "MVP", "득점왕", ...]
  description: string | null          // 대회 소개 텍스트
  bannerImageUrl: string | null       // 대회 배너 이미지
```

### 2-3. 대회 상태 흐름

```
  draft ──→ registration_open ──→ registration_closed ──→ in_progress ──→ completed
   │              │                                           │              │
   │              ▼                                           ▼              ▼
   │         (팀 참가 신청/승인)                          (경기 진행 중)    (시상/통계 확정)
   │                                                         │
   └──→ cancelled                                        suspended
                                                          (일시 중단)
```

| 상태 | 설명 | 허용 액션 |
|---|---|---|
| `draft` | 초안 작성 중 | 설정 수정, 삭제 |
| `registration_open` | 참가 접수 중 | 팀 참가 신청/승인/거절, 설정 수정 |
| `registration_closed` | 참가 마감 | 대진표 생성, 일정 배정 |
| `in_progress` | 대회 진행 중 | 경기 기록, 결과 입력, 라인업 편집 |
| `completed` | 대회 종료 | 통계 조회, 시상 확정 |
| `suspended` | 일시 중단 (우천 등) | 재개 또는 취소 |
| `cancelled` | 취소됨 | 조회만 가능 |

## 3. 팀 참가 관리

### 3-1. 참가 신청 플로우

**공개 대회 (선수 → 총무):**

```
팀 대표가 /league 페이지에서 대회 발견
        │
        ▼
  [참가 신청] 버튼 클릭
        │
        ▼
  참가 신청 폼 작성
  - 팀명, 대표자, 연락처
  - 예상 참가 인원
  - 유니폼 색상 (메인/보조)
  - 참가비 납부 확인 (선택)
        │
        ▼
  대회 운영자에게 알림 발송
        │
        ├──→ 승인 → 참가 확정 → 팀 대표에게 알림
        └──→ 거절 → 사유와 함께 팀 대표에게 알림
```

**비공개 대회 (총무 → 팀):**

```
총무가 /manage/league에서 초대 링크 생성
        │
        ▼
  카카오톡으로 초대 링크 공유
        │
        ▼
  팀 대표가 링크 클릭 → 자동 참가 승인
```

### 3-2. 참가팀 관리 화면

**라우트:** `/manage/league/{leagueId}/teams`

| 기능 | 설명 |
|---|---|
| 참가팀 목록 | 팀명, 대표자, 인원, 유니폼 색상, 신청일, 상태(대기/승인/거절) |
| 일괄 승인/거절 | 체크박스 선택 후 일괄 처리 |
| 시드 배정 | 토너먼트 시 시드 번호 수동 배정 또는 랜덤 배정 |
| 참가비 현황 | 팀별 참가비 납부 상태 추적 (재정 Lambda 연동) |
| 초대 링크 | 비공개 대회용 1회성/다회성 초대 링크 생성 |

## 4. 대진표 시스템

### 4-1. 대진표 자동 생성 로직

**토너먼트 (엘리미네이션):**

```
입력: 참가팀 목록 + 시드 배정
        │
        ▼
  팀 수 → 가장 가까운 2^n으로 올림 (부전승 처리)
        │
        ▼
  시드 배정 기반 대진 매칭
  - 1시드 vs 최하위 시드 (반대편 브래킷)
  - 같은 지역 팀 분리 배치 (가능한 경우)
        │
        ▼
  1라운드 매치 자동 생성
  - 부전승 팀은 자동 2라운드 진출
        │
        ▼
  라운드 완료 시 다음 라운드 자동 생성
  (기존 league Lambda 로직 유지)
```

**시드 배정 대진 예시 (8팀):**

```
1라운드                준결승              결승

  #1 시드 ─┐
           ├── 승자 ─┐
  #8 시드 ─┘         │
                     ├── 승자 ─┐
  #4 시드 ─┐         │         │
           ├── 승자 ─┘         │
  #5 시드 ─┘                   │
                               ├── 🏆 우승
  #2 시드 ─┐                   │
           ├── 승자 ─┐         │
  #7 시드 ─┘         │         │
                     ├── 승자 ─┘
  #3 시드 ─┐         │
           ├── 승자 ─┘
  #6 시드 ─┘
```

**리그 (라운드 로빈):**

```
입력: 참가팀 목록
        │
        ▼
  모든 팀 조합으로 매치 생성
  - N팀 → N*(N-1)/2 경기 (편도)
  - 또는 N*(N-1) 경기 (홈&어웨이)
        │
        ▼
  라운드 분배 (한 라운드에 팀당 최대 1경기)
        │
        ▼
  일정 자동 배정 (시작일~종료일 내 균등 분배)
```

### 4-3. 대진표 인터랙션 상세

| 기능 | PC | 모바일 |

### 4-4. 대진표 렌더링 기술 명세

```
데이터 변환 파이프라인:

league.matches (API 응답)
    │
    ▼
matchesByRound = groupBy(matches, 'round')
    │
    ▼
bracketTree = buildTree(matchesByRound)
  - 각 매치에 nextMatchId 연결
  - 부전승(bye) 노드 삽입
  - 승자 진출 경로 추적
    │
    ▼
렌더링 (2가지 옵션):

  옵션 A: CSS Grid 기반
  - 라운드별 column, 매치별 row
  - CSS line으로 연결선
  - 장점: 반응형 용이, 접근성 우수
```

## 5. 경기 운영 상세

### 5-1. 경기 상태 흐름 (기존 확장)

```
기존:
  proposed → scheduled → homeSubmitted/awaySubmitted → bothSubmitted/disputed → confirmed

확장:
  proposed → scheduled → pregame → live → ended → homeSubmitted/awaySubmitted → bothSubmitted → confirmed
                           │                │
                           │                └→ disputed → confirmed (운영자 중재)
                           │
                           └→ postponed → rescheduled → (scheduled로 복귀)
                              cancelled
```

| 상태 | 설명 | 트리거 |
|---|---|---|
| `pregame` (신규) | 경기 시작 전 (라인업 확정, 타임테이블 공개) | 경기 시작 1시간 전 자동, 또는 총무 수동 |
| `live` (신규) | 경기 진행 중 (실시간 이벤트 기록 가능) | 총무가 "경기 시작" 버튼 클릭 |
| `ended` (신규) | 경기 종료 (이벤트 기록 마감, 결과 제출 대기) | 총무가 "경기 종료" 버튼 클릭 |
| `postponed` (신규) | 연기 (우천, 부상 등) | 총무가 연기 처리 + 사유 입력 |

### 6-2. 통계 집계 로직

기존 `GET /league/{id}/stats` 엔드포인트를 확장한다.

```
집계 대상 이벤트 → pg-match-events 테이블 스캔

득점 순위:
  SELECT playerId, playerName, teamId,
         COUNT(goal) as goals,
         COUNT(assist) as assists
  FROM match-events
  WHERE leagueId = :id AND eventType = 'goal'
  GROUP BY playerId
  ORDER BY goals DESC

카드 누적:
  SELECT playerId, playerName,
         SUM(yellow_card) as yellows,
         SUM(red_card) as reds
  FROM match-events
  WHERE leagueId = :id AND eventType IN ('yellow_card', 'red_card')
  GROUP BY playerId

징계 판단:
  IF yellows >= league.rules.yellowCardAccumulation
    → 다음 경기 출장 정지
  IF reds >= 1
    → league.rules.redCardSuspension 경기 출장 정지
```

| D-1 공지 | `[D-1] {대회명} 결전의 날!` | 집결 시간/장소, 지급품 | 출전 쿼터 확인 | 타임테이블 보기 |
| 경기 결과 | `[결과] {홈팀} {스코어} {원정팀}` | MOM, 주요 득점자 | 상세 스탯 보기 | 다음 경기 일정 |
| 대회 개막 | `🏆 {대회명} 개막!` | 참가팀 수, 첫 경기 일정 | 대진표 보기 | 참가팀 목록 |
| 우승 축하 | `🏆 {팀명} 우승!` | 최종 스코어, MVP | 대회 통계 보기 | 공유하기 |

### 7-3. OG(Open Graph) 이미지 자동 생성

카카오톡 메시지 카드의 썸네일 이미지를 동적으로 생성한다.

```
API: GET /og/vote/{scheduleId}.png
API: GET /og/result/{matchId}.png
API: GET /og/bracket/{leagueId}.png

생성 방식:
  - Lambda에서 HTML 템플릿 렌더링 → Puppeteer/Playwright로 스크린샷
  - 또는 @vercel/og 스타일의 Satori 기반 SVG → PNG 변환
  - S3에 캐싱 (같은 요청 반복 시 캐시 반환)

이미지 사이즈: 800 x 400px (카카오 권장 비율 2:1)
```

## 9. 신규 DynamoDB 테이블

| 테이블명 | PK | SK | GSI | 용도 |
|---|---|---|---|---|
| `pg-vote-responses` | `scheduleId` | `userId` | — | 참석 투표 |
| `pg-match-lineups` | `matchId` | `quarter#{n}` | — | 쿼터별 라인업 |
| `pg-match-timetable` | `matchId` | `order#{n}` | — | 당일 타임테이블 |
| `pg-match-events` | `matchId` | `event#{ts}#{type}` | `GSI-playerId` (playerId → matchId) | 경기 이벤트 |
| `pg-match-mom` | `matchId` | `voterId` | — | MOM 투표 |
| `pg-league-registrations` | `leagueId` | `teamId` | `GSI-status` (status → leagueId) | 참가 신청 |

**CDK GSI 배포 주의:** DynamoDB GSI는 한 번의 CloudFormation 업데이트에 1개씩만 추가 가능. `pg-match-events`의 `GSI-playerId`와 `pg-league-registrations`의 `GSI-status`는 별도 배포 사이클로 추가할 것.

### 11-3. 주의 사항 (기존 gotcha 준수)

- `useSearchParams`를 사용하는 신규 페이지는 반드시 `<Suspense>` 래핑
- 모든 `useState`/`useEffect`는 조건부 return 이전에 선언
- React Compiler 규칙 준수: ref 접근은 useEffect 내에서만
- 빌드 시 `npm run lint` 명시적 실행 (ignoreBuildErrors 주의)
- CDK GSI 추가는 1개씩 별도 배포
- S3 배포 시 `--exclude "uploads/*"` 유지
