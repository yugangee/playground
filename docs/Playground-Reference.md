## 1. 프로젝트 한 줄 요약

**Playground는 아마추어 스포츠 동호회의 운영과 참여를 하나의 앱에서 해결하는 수직 통합 플랫폼이다.** 현재 축구(football)에 집중하되, 데이터 구조는 멀티스포츠 확장이 가능하게 설계한다.

## 3. 핵심 의사결정 사항 (확정됨)

이전 대화와 KB_05를 통해 다음 사항이 확정되었다.

### 3-1. 전략적 결정

| 결정 사항 | 내용 | 근거 |
|----------|------|------|
| **카톡 공존 전략** | 카톡 대체가 아니라 카톡과 공존하며 점진적 유입 | 30명 중 15명이 앱 설치를 거부하는 현실. 카톡 습관을 깨지 않는 게 현실적 |
| **플랫폼: PWA 우선** | 반응형 웹(PWA) → 앱스토어 래핑 → 필요 시 네이티브 | 회원 불만 1위(앱 설치)를 원천 제거. 알림톡 브릿지와 자연스러운 연결 |
| **알림톡 브릿지** | 카카오 알림톡을 Phase 1 핵심 알림 채널로 채택 | 건당 8~15원(월 500원 미만), 개발 1~2주, 검증된 인프라 |
| **캘린더 필수 확정** | 일정·참석 메뉴의 기본 뷰로 캘린더 채택 | 회원 불만 3위 + 총무 3대 업무. 참석 응답과 통합 시 가치 극대화 |
| **킬러 피처 1순위** | 알림톡 + 원클릭 참석 응답 | 총무 1위 고통(전화 돌리기) + 회원 2위 불만(응답 귀찮음) 동시 해결 |

### 3-2. 기각된 대안

| 대안 | 기각 이유 |
|------|----------|
| 네이티브 앱 우선 출시 | 설치 장벽이 최대 허들. PWA로 이 문제를 원천 제거해야 함 |
| 카톡 완전 대체 | 비현실적. 30명 중 절반이 이동하지 않으면 서비스가 죽음 |
| 카톡만 사용 (Playground 없이) | 데이터 축적 불가. 선순환 루프가 작동하지 않음 |
| 회비 관리를 1순위로 | 참석 데이터가 먼저 쌓여야 정산·회비가 의미 있음 |
| 리스트 뷰만 제공 | 시각적 파악 불가. 캘린더가 있어야 "보기→응답" 1초 |

## 4. 사용자 리서치 핵심 발견

이전 대화에서 커뮤니티 분석, 서비스 사례, 정부 통계를 종합한 실증 리서치를 수행했다.

### 4-1. 시장 데이터

- 체육 동호회 가입률: 전 국민의 11.3% (2023 국민생활체육조사)
- 동호회 가입 종목 1위: 축구·풋살 22.9% → 축구 우선 전략의 근거
- 2위 배드민턴(12.3%), 3위 탁구(10.5%)
- 생활체육 미참여율 32.2% (최근 5년 최대치) → 이탈 방지 필요

### 4-2. 총무 고통점 TOP 5

총무의 주당 행정 시간은 약 **230분(3.8시간)**으로 추정된다.

| 순위 | 고통점 | 심각도 | 주요 소요 시간 | Playground 해법 |
|:---:|--------|:---:|:---:|----------------|
| 1 | 참석 인원 확인 (전화 돌리기) | 95/100 | 매주 60분+ | 자동 리마인드 + 원클릭 응답 + 최소인원 미달 경고 |
| 2 | 회비 독촉 & 정산 | 88/100 | 매월 45분 | 자동 미납 알림 + 실시간 지출 공개 |
| 3 | 구장 예약 & 변경 관리 | 75/100 | 매주 30분 | 참석 현황 연동 구장 추천 + 캘린더 |
| 4 | 공지 전달 & 확인 | 70/100 | 매주 20분 | 공지 전용 채널 + 읽음 확인 |
| 5 | 인원 관리 & 분쟁 조정 | 65/100 | 매 경기 15분 | 출전 시간 자동 기록 + 공정 배분 데이터 |

### 4-3. 회원 불만 TOP 5

| 순위 | 불만 | 귀찮음 점수 | Playground 해법 |
|:---:|------|:---:|----------------|
| 1 | 새 앱 설치 | 95/100 | PWA + 알림톡 브릿지 (설치 불필요) |
| 2 | 참석 여부 응답 | 90/100 | 알림톡에서 버튼 하나로 응답 (1초) |
| 3 | 경기 일정 파악 | 82/100 | 캘린더 뷰 + 내비 연동 |
| 4 | 회비 납부 확인 | 75/100 | 납부 현황 대시보드 + 자동 알림 |
| 5 | 사진·영상 찾기 | 55/100 | 경기별 자동 분류 아카이브 (Phase 2) |

### 4-4. 핵심 인사이트 3가지

**인사이트 1**: 총무의 1위 고통(참석 확인 전화)과 회원의 2위 불만(응답 귀찮음)은 **같은 문제의 양면**이다. 이 하나만 해결하면 Playground의 존재 이유가 생긴다.

**인사이트 2**: 최대 허들은 "앱 설치" 자체다. **카톡 알림톡으로 참석 응답을 받고, 캘린더 링크를 공유하는 것**이 Phase 1의 최우선 과제다.

**인사이트 3**: 선순환 루프가 검증되었다. 총무 1회 입력 → 알림톡 자동 발송 → 회원 1초 응답 → 실시간 집계 → 총무 행정시간 50%↓ → 더 자주 업데이트 → 회원 점진적 앱 설치.

## 5. 서비스 로드맵 전체 구조

### 5-1. 3단계 Phase

| Phase | 목표 | 핵심 질문 | 주요 기능 |
|:---:|------|----------|----------|
| **Phase 1 ★☆☆** | 카톡+엑셀 대체 | "카톡 대신 이 앱 쓸 이유가 있나?" | 팀·회원, 일정·참석(캘린더), 회비, 소통(채팅), 알림톡 브릿지 |
| **Phase 2 ★★☆** | 자발적 앱 접속 | "내 기록이 쌓이는 곳이 어디지?" | 경기 기록·통계, MVP 투표, PIS, 미디어 아카이브, AI 리포트 |
| **Phase 3 ★★★** | 생태계 확장 | "경기 없는 날에도 왜 이 앱을 열지?" | 전술 보드, 재무 고도화, 외부 리그 연동, AI 하이라이트, 마켓 |

**불변 원칙**: Phase 순서를 뒤집지 않는다. Phase 1 데이터가 쌓여야 Phase 2가 의미 있고, Phase 2 콘텐츠가 풍부해야 Phase 3 생태계가 작동한다.

## 7. 카카오톡 알림톡 브릿지 상세

Phase 1의 핵심 기술 인프라다.

### 7-1. 구현 구조

```
Playground 서버 → 알림톡 중계사(솔라피/알리고/바로빌) API
    → 회원 전화번호로 카카오 알림톡 발송
        → 회원이 카톡에서 "참석/불참" 버튼 클릭
            → Playground 모바일 웹 페이지 → 응답 기록
```

### 7-2. 비용과 개발 난이도

- 알림톡 건당: 약 8~15원
- 30명 팀 × 주 1회: 월 약 500원 미만 (사실상 무시 가능)
- SMS fallback: 건당 약 20원
- 개발: 백엔드 1명, 1~2주

### 7-3. 제약

- 정보성 메시지만 허용 (경기 안내 + 참석 응답은 정보성이므로 OK)
- 메시지 템플릿 사전 심사 필요 (1~3영업일, 1회 통과 시 계속 사용)
- 회원 전화번호 필수 (총무가 이미 보유)
- 카톡 단체방 자동 메시지는 불가 → 공유 링크 카드로 대응

### 7-4. 멀티채널 우선순위

| 순위 | 채널 | 연동 방식 |
|:---:|------|----------|
| 1순위 | 카카오 알림톡 | API 자동 발송 + 웹 응답 링크 |
| 1순위 | SMS/LMS | 알림톡 실패 시 자동 fallback |
| 1순위 | 공유 링크 카드 | 카톡/밴드/카페에 붙여넣기용 |
| 2순위 | 앱 푸시 | FCM/APNs (Phase 1.5+) |
| 3순위 | 네이버 밴드 | Open API 자동 포스팅 (안정화 후) |
| 대안 없음 | 다음 카페 | 공식 API 미제공, 수동 링크만 가능 |

### 7-5. 듀얼 플랫폼 운영 원칙

Playground가 **원본 데이터의 단일 소스(Single Source of Truth)**이고, 외부 채널은 알림과 응답 수집의 파이프라인 역할만 한다. 총무는 Playground에서 1회 입력, 회원은 카톡 습관을 깨지 않고 응답, 데이터는 Playground에 축적.

## 10. 데이터 모델 핵심

### 10-1. 멀티스포츠 설계 원칙

모든 데이터 모델에 `sport_type` 필드 포함. 종목 공통 모듈(팀·일정·회비·게시판·미디어)과 종목 특화 모듈(포지션·기록 스키마·전술·성과 지표)을 분리.

### 10-2. 주요 엔터티

```
Team ─── sport_type: enum
 ├── Member (1:N) ─── role: COACH | MANAGER | MEMBER | GUEST
 ├── Schedule (1:N) ─── type: MATCH | TRAINING | EVENT
 │    └── Attendance (1:N)
 ├── Match (1:N)
 │    ├── MatchEvent (1:N) ─── goal, assist, card, substitution
 │    ├── MatchQuarter (1:N)
 │    └── MvpVote (1:N)
 ├── Finance (1:N)
 ├── Post (1:N)
 ├── Media (1:N)
 └── Season (1:N)
```

### 10-3. 핵심 공식

**PIS (Performance Impact Score)**:
```
PIS = (G × α) + (A × β) + (M × γ) + (T/90 × δ) + (V × ε)
```
포지션별 가중치 차등 (공격수: 득점↑ / 수비수: 출전·동료평가↑ / 골키퍼: MVP·동료평가↑)

**Runway (재무 건전성)**:
```
Runway(개월) = C_res / (E_avg − I_fixed)
```
≥6개월: 안정 / 3~6: 주의 / <3: 경고

### 10-4. 등급 시스템 (검토 중)

이전 대화에서 Tier 시스템(B→S→A→SP→P) 기획안을 검토했다. PIS와의 관계 정리가 필요하며, 시즌 리셋/감쇠 규칙 추가, Phase 2 이후 배치가 권장된다. PIS = "얼마나 잘 했나", Tier = "얼마나 꾸준히 이겼나"로 역할 분리 방향이 논의됨.

## 11. 선순환 구조 (Feedback Loop)

검증된 Phase 1 선순환 루프:

```
[총무] Playground에서 일정 1회 입력
    ↓
[시스템] 카카오 알림톡 자동 발송 + 카톡방 공유 링크 생성
    ↓
[회원] 카톡에서 버튼 1개 클릭 → 참석/불참 응답 (1초)
    ↓
[시스템] Playground 캘린더에 참석 현황 실시간 집계
    ↓
[총무] 전화 돌릴 필요 없음 → 행정시간 50%↓
    ↓
[총무] 부담 감소 → 더 자주 일정 등록·업데이트
    ↓
[회원] 콘텐츠 풍부 → "이 서비스 편하네" → 점진적 앱 설치
    ↓
[데이터] 참석·일정 데이터 축적 → Phase 2 기록·통계 기반 마련
```

장기적으로: Phase 1 데이터 축적 → Phase 2 콘텐츠 풍부 → Phase 3 생태계 확장. 이 순서를 뒤집으면 빈 데이터 위의 기능이 된다.
