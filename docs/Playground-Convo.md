## 세션 1: Phase 1 기능 갭 분석

| 항목 | 내용 |
|------|------|
| **일시** | 2026.02.23 (오후) |
| **트랜스크립트** | `2026-02-23-14-55-40-phase1-feature-gaps-analysis.txt` |

### 사용자 요청

사용자가 Playground의 **개인 등급 시스템(Tier System) 기획안 이미지**를 첨부하고 평가를 요청했다. 이어서 "1단계에서 가입자, 사용자를 늘리는 게 가장 큰 관건"이라며 Phase 1 기능 강화 제안을 요청했다.

> "초기 1단계 기능을 더 강화할 걸 추천해 주세요. 아주 기본적이면서 핵심적인 서비스 중에 빠진 건 없는지, 추가하면 좋은 게 있는지 알아보고 제안해 주세요."

### Claude 분석 및 응답

**등급 시스템 평가**: 참여 자체에 +3점을 주는 구조는 동호회 핵심 가치(출석 유인)를 잘 반영. 연승 보너스(+1/+2/+3)도 깔끔한 규칙. 다만 등급 시스템은 Phase 2 영역으로, Phase 1에서는 데이터 축적이 우선.

**Phase 1 핵심 기능 갭 5가지 도출**:

| 순위 | 빠진 기능 | 왜 빠지면 안 되는가 |
|:---:|----------|-------------------|
| 1 | **카카오톡 알림 브릿지** | 회원이 앱을 안 깔아도 카톡에서 참석 응답 가능. 이게 없으면 총무가 이중 작업 |
| 2 | **경기일 자동 정산 (1/N 더치페이)** | 매주 구장비·음료비를 참석 인원으로 나누는 계산을 총무가 수동으로 함 |
| 3 | **구장 정보 DB + 즐겨찾기** | 일정 등록 시 구장 선택 → 위치·대관료 자동 연동 → 내비 링크 발송 |
| 4 | **패키지 위험 경고** | 최소 인원 미달 시 자동 경고 → 총무의 "몰수패 공포" 해결 |
| 5 | **카카오톡 공존 설계** | "카톡을 버려라"가 아닌 "카톡에서 Playground로 자연스럽게 유도" |

**핵심 관건 정의**: Phase 1의 진짜 목표는 "총무가 편해지는 것"이 아니라 **"회원이 저항 없이 올라타는 것"**. 30명 중 절반도 앱을 설치하지 않는 현실에서, 앱 설치 없이 참여 가능한 구조가 선행 필수.

### 이 세션의 핵심 결정

- ✅ **카카오톡 알림 브릿지**가 Phase 1 최우선 추가 기능으로 부상
- ✅ "카톡 대체"가 아닌 **"카톡 공존"** 방향으로 전략 전환
- ✅ 전환 장벽 제거가 Phase 1의 최대 관건임을 확인

## 세션 2: 카카오톡 브릿지 기술 검증

| 항목 | 내용 |
|------|------|
| **일시** | 2026.02.23 (오후, 세션 1 직후) |
| **트랜스크립트** | `2026-02-23-14-57-04-kakao-bridge-feasibility-multiplatform.txt` |

### 사용자 요청

> "카카오톡 브릿지가 어렵지 않나요? AI로 가능한가요? 둘이 동시 호환도 가능한가요? 왔다갔다. 카톡 뿐만 아니라 다음 카페, 밴드 등도 호환이 필요하지 않나요"

### Claude 분석 및 응답

**웹 검색 수행**: 카카오 알림톡 비즈메시지 API, 솔라피/알리고 중계사, 네이버 밴드 Open API, 다음 카페 API 현황 조사.

#### 1. 알림톡 브릿지 — 기술적으로 검증 완료

- **구현 구조**: Playground 서버 → 알림톡 중계사(솔라피/알리고/바로빌) API 호출 → 회원 전화번호로 카카오 알림톡 발송 → 회원이 카톡에서 "참석/불참" 버튼 클릭 → Playground 모바일 웹 페이지로 이동 → 응답 기록
- **비용**: 건당 8~15원, 30명 팀 주 1회 발송 시 월 500원 미만 (사실상 무시 가능)
- **제약**: 정보성 메시지만 허용 (경기 안내 + 참석 응답은 정보성이므로 허용), 템플릿 사전 심사 1~3영업일
- **개발 난이도**: 백엔드 개발자 1명 1~2주 소요. 중계사가 REST API + SDK 제공

#### 2. AI 활용 가능성

알림톡 자체는 메시징 API이지 AI가 아님. 다만 AI 결합 시 자연어 입력("이번 주 토요일 2시 월드컵경기장") → 날짜·시간·장소 자동 추출 → 일정 등록 + 알림톡 발송 자동화 가능. 이 기능은 Phase 1 이후 고도화 단계에서 적용.

#### 3. 카톡+Playground 동시 호환 — 듀얼 플랫폼 운영

**핵심 원칙**: "카톡을 버리고 Playground로 오라"가 아니라 **양쪽이 공존하는 구조**.

- 총무 → Playground에서 일정 등록 → 자동으로 두 곳에 발신:
  1. 카카오 알림톡으로 회원 개인에게 발송
  2. 카톡 단체방에 공유 가능한 링크 카드 생성 (총무가 한 번 붙여넣기)
- 회원 → 카톡에서 링크 클릭 → Playground 모바일 웹에서 참석/불참 응답 → 응답 결과가 Playground에 자동 집계
- Playground가 **원본 데이터의 단일 소스(Single Source of Truth)**, 외부 채널은 파이프라인 역할

#### 4. 멀티채널 호환성 정리

| 채널 | 연동 방식 | 난이도 | 우선순위 |
|------|----------|:---:|:---:|
| **카카오 알림톡** | API 자동 발송 + 웹 응답 링크 | 중 | **1순위** |
| **SMS/LMS** | 알림톡 실패 시 자동 대체 발송 (fallback) | 저 | **1순위** (fallback) |
| **공유 링크 카드** | 카톡/밴드/카페에 붙여넣기용 | 저 | **1순위** |
| **앱 푸시** | FCM/APNs | 중 | **2순위** |
| **네이버 밴드** | Open API 자동 포스팅 가능 | 중 | **3순위** |
| **다음 카페** | 공식 API 미제공. 공유 링크 수동 붙여넣기만 가능 | - | 대안 없음 |
| **카톡 단체방 자동 메시지** | 불가. 카카오가 외부 서비스의 단체방 직접 전송 미허용 | - | - |

### 이 세션의 핵심 결정

- ✅ **알림톡 브릿지** 기술적 실현 가능성 확인 (비용·개발 난이도 모두 낮음)
- ✅ **듀얼 플랫폼 운영** 원칙 확정 ("카톡 대체"가 아닌 "카톡 공존")
- ✅ 1순위(알림톡+SMS+공유링크)만 구현해도 Phase 1에서 100% 도달률 확보 가능
- ✅ 네이버 밴드는 3순위, 다음 카페는 공식 연동 불가

#### 회원 불만 TOP 5

| 순위 | 불만 | 귀찮음 | 핵심 | Playground 해법 |
|:---:|------|:---:|------|------------|
| 1 | **새 앱 설치** | 95/100 | 30명 중 절반 설치 거부 | PWA + 알림톡 브릿지 |
| 2 | **참석 여부 응답** | 90/100 | 카톡 투표 무시 → 총무 전화 | 알림톡 버튼 1개 (1초) |
| 3 | **경기 일정 파악** | 82/100 | 카톡방 스크롤 고통 | 캘린더 뷰 + 내비 연동 |
| 4 | **회비 납부** | 75/100 | 얼마, 냈는지 모름 | 납부 현황 + 자동 알림 |
| 5 | **사진·영상 찾기** | 55/100 | 2주 지나면 만료 | 경기별 아카이브 (Phase 2) |

#### 핵심 인사이트 3가지

1. **총무 1위 고통 = 회원 2위 귀찮음** (참석 확인 = 참석 응답). 같은 문제의 양면. 이 하나만 해결하면 Playground의 존재 이유가 생긴다.
2. **최대 허들은 "앱 설치" 자체**. 아무리 좋은 기능도 30명 중 15명이 설치 거부하면 무의미. 카카오톡 알림톡 브릿지가 선행 필수.
3. **선순환 루프 검증됨**: 총무 1회 입력 → 알림톡 발송 → 회원 1초 응답 → 실시간 집계 → 총무 행정시간 50%↓ → 더 자주 업데이트 → 회원 점진적 유입.

#### 캘린더 기능 확정

캘린더는 "있으면 좋겠다"가 아니라 **Phase 1 필수 기능**으로 확정.

| 근거 | 상세 |
|------|------|
| 회원 불만 3위 | 일정 파악 어려움 |
| 총무 3대 업무 | 일정 관리 = 구장 예약과 직결 |
| 시각적 구분 | 경기(🔴)·훈련(🔵)·회식(🟢) 색상 구분 |
| 응답률 단축 | 캘린더 안에 참석 버튼 내장 → "보기→응답" 1초 |

**Phase 1 캘린더 필수 기능**: 월간/주간 뷰 토글, 이벤트 유형 색상 구분, 날짜 탭 → 상세 펼침, 참석/불참 버튼 내장, 구글캘린더·아이폰캘린더 동기화(ics export), 리마인드 연동.

### 이 세션의 핵심 결정

- ✅ **실증 데이터 기반** 총무 고통점 5개 + 회원 불만 5개 도출
- ✅ 총무 주당 행정시간 230분(3.8시간) 추정 — 팀 해체의 핵심 원인
- ✅ **캘린더를 Phase 1 필수 기능으로 확정** (일정·참석 메뉴의 기본 뷰)
- ✅ **킬러 피처 우선순위 확정**: 1순위 알림톡+원클릭, 2순위 캘린더, 3순위 회비 대시보드
- ✅ 선순환 루프 모델 검증 완료

## 세션 4: 플랫폼 전략 — 웹 vs 앱

| 항목 | 내용 |
|------|------|
| **일시** | 2026.02.23 (야간) |
| **트랜스크립트** | `2026-02-23-23-22-11-platform-strategy-web-pwa-first.txt` |

### 사용자 요청

> "일단 웹으로 시작하는데 바로 앱으로 가는 게 낫나요?"

### Claude 분석 및 응답

**웹 검색 수행**: PWA 기술 현황(iOS 16.4+ 웹 푸시), React Native/Flutter 웹+앱 전략, 한국 시장 PWA 사례 조사.

**결론**: "웹(PWA) 먼저 → 검증 후 앱 래핑 → 필요 시 네이티브" 3단계 전략이 최적.

#### PWA 우선의 근거 6가지

| # | 근거 | 상세 |
|---|------|------|
